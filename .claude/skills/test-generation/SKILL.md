# テスト生成

DSL仕様書と構成の知識に基づき、生成されたリストコンテナの正しさを
検証するテストコードを生成するスキル。
従来のGPにおけるADT仕様（事前条件・事後条件・不変条件）を、
実行可能なテストとして表現する。

## 前提確認

- `docs/gp/09-configuration-knowledge.md` が存在するか確認する
- テスト対象の生成コードが `generated/<言語>/` に存在するか確認する
- 対応するDSL仕様書が `docs/gp/08-dsl-examples/` に存在するか確認する

## 引数

- テスト対象のDSL仕様書のパス、または生成コードのディレクトリ
- 省略時は、既存の生成コードを検出して対話的に選択する

## 手順

### ステップ1: 構成の特定

1. DSL仕様書をバリデーション・構成解決する
2. 構成解決結果から、テストすべき性質を特定する:
   - 共通性質（すべての構成で成立）
   - 構成固有の性質（選択されたフィーチャーに依存）

### ステップ2: 共通性質のテスト生成

すべての構成で成立する代数的性質をテストとして生成する:

```
P1: length(new()) = 0
    → 新規リストの要素数は0

P2: isEmpty(new()) = true
    → 新規リストは空

P3: length(insert(l, e)) = length(l) + 1
    → 挿入は要素数を1増やす

P4: isEmpty(insert(l, e)) = false
    → 挿入後は空ではない

P5: find(insert(l, e), e) ≠ null
    → 挿入した要素は検索で見つかる

P6: ¬isEmpty(l) ⟹ length(remove(l)) = length(l) - 1
    → 非空リストからの削除は要素数を1減らす

P7: isEmpty(l) ⟹ remove(l) returns false
    → 空リストからの削除は失敗する
```

### ステップ3: 構成固有のテスト生成

選択されたフィーチャーに応じた追加テストを生成する:

#### Ownership = Copy の場合
```
PC1: 挿入後に元の値を変更しても、リスト内の値は変化しない（コピーの独立性）
```

#### Ownership = ExternalReference の場合
```
PE1: 挿入後に元の値を変更すると、リスト経由でも変更が観測される（参照の共有性）
```

#### Ownership = OwnedReference の場合
```
PO1: destroy後、全要素が解放される（所有権による解放責任）
```

#### LengthCounter = 有効 の場合
```
PL1: length() の返り値はO(1)で取得される（カウンタの存在確認）
PL2: 複数回のinsert/removeの後もlength()は正確（カウンタ整合性）
```

#### Tracing = 有効 の場合
```
PT1: insert/remove/find/destroy の各操作がトレースログに記録される
PT2: トレースログの操作順序は実際の操作順序と一致する
```

### ステップ4: テストコードの出力

生成コードと同じ言語でテストを出力する。

出力先: 生成コードと同じディレクトリ、または言語の慣習に従ったテストディレクトリ。

テストフレームワーク:
- C++: 素のassertマクロ、またはプロジェクトにテストFWがあればそれを使用
- Java: JUnit
- Go: testing パッケージ
- PHP: PHPUnit
- Python: pytest
- Rust: #[test]

### ステップ5: テストの実行と検証

生成したテストを実行し、全テストがパスすることを確認する。
失敗するテストがあれば:
1. 生成コードのバグか、テストの期待値の誤りかを判断する
2. 生成コードのバグの場合は修正する
3. テストの期待値の誤りの場合はテストを修正する
