# フィーチャー間の関係パターン集

## requires（依存）
フィーチャーAを有効にするにはフィーチャーBも有効でなければならない。
Bを無効にする変更を行う場合、Aも無効にする必要がある。
例: OAuth認証 → HTTPS通信

## excludes（排他）
フィーチャーAとフィーチャーBは同時に有効にできない。
Aを追加する変更を行う場合、Bを先に無効にする必要がある。
例: リアルタイム同期 ⊗ バッチ処理（同一データに対して）

## impacts（影響）
フィーチャーAの変更がフィーチャーBに影響を与える可能性がある。
壊れるとは限らないが、確認が必要。
例: データモデルの変更 → レポート機能の出力確認

## change-chain（変更波及チェーン）
フィーチャーAの変更がB、BからCへと波及するパス。
変更計画時にこのチェーン全体を確認する必要がある。
例: 認証方式変更 → セッション管理変更 → API認可変更

## recommends（推奨）
フィーチャーAを有効にするなら、フィーチャーBも有効にすることが望ましい。
強制ではないが、理由を理解した上で判断すべき。
例: 全文検索 → インデックスキャッシュを推奨

## parameter-constraint（パラメータ制約）
フィーチャーAのパラメータ値がフィーチャーBの有効性や設定に影響する。
例: 同時接続ユーザー数 > 1000 → キャッシュ機能が必須
