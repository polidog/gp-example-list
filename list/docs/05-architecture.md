# アーキテクチャ設計: リストコンテナ

## アーキテクチャスタイル

### 選定したスタイル
静的構成によるコードフラグメント合成（Static Configuration with Code Fragment Composition）

### 選定理由
- AIによるコード生成が前提であり、実行時ではなく**生成時に可変性を解決**する
- リストコンテナは単一モジュールとして完結するため、プラグイン機構やDIコンテナは不要
- フィーチャー選択に基づいて必要なコードフラグメントを組み合わせ、1つのソースファイルを生成する方式が最もシンプル
- 言語非依存のため、言語固有のランタイム抽象化（インターフェース、トレイト等）に依存しない設計が必要

## 可変ポイントの実現方式

| 可変ポイント | 変化のタイプ | 実現方式 | 説明 |
|-------------|------------|---------|------|
| ElementType | パラメータ型 | 型パラメータ置換 | 言語のジェネリクス機構または具体型をテンプレートに埋め込む |
| Storage（ArrayBased / LinkedList） | 差し替え型 | コードフラグメント選択 | データ構造ごとに操作の実装フラグメントを切り替える |
| InitialCapacity | パラメータ型 | 定数値埋め込み | 生成コード内の定数として設定値を埋め込む |
| GrowthStrategy | 差し替え型 | コードフラグメント選択 | 拡張ロジックのフラグメントを切り替える |
| Direction（LinkedList） | 差し替え型 | コードフラグメント選択 | ノード構造体とポインタ操作のフラグメントを切り替える |
| OptionalOperations | 追加型 | コードフラグメント追加 | 選択された操作のフラグメントを生成コードに追加する |
| SortAlgorithm | 差し替え型 | コードフラグメント選択 | ソートアルゴリズムの実装フラグメントを切り替える |
| Iteration Direction | 差し替え型 | コードフラグメント選択 | イテレータの走査方向フラグメントを切り替える |

## モジュール構成

```
生成されるリストコンテナ（1ファイル）
├── 型定義セクション
│   ├── リスト構造体（共通）
│   └── 内部データ構造（可変: ArrayBased / LinkedList）
├── コンストラクタセクション
│   └── 初期化ロジック（可変: Storage + パラメータに依存）
├── 基本操作セクション（共通）
│   ├── Add
│   ├── Get
│   └── Size
├── オプション操作セクション（可変: 選択されたもののみ生成）
│   ├── Remove
│   ├── Insert
│   ├── Search（LinearSearch / Contains）
│   └── Sort（InsertionSort / MergeSort）
├── イテレーションセクション（可変: 選択された場合のみ生成）
│   └── Iterator（Forward / Reverse）
└── コントラクト適合セクション（可変: 言語×フィーチャーから導出）
    └── 言語ネイティブのinterface/protocol実装メソッド
```

### 各セクションの責務

| セクション | 責務 | 共通/可変 |
|-----------|------|----------|
| 型定義 | リスト構造体と内部データ構造の定義 | 可変（Storageに依存） |
| コンストラクタ | リストの初期化 | 可変（Storage + パラメータに依存） |
| 基本操作 | Add, Get, Size の実装 | 共通（インターフェースは固定、実装はStorageに依存） |
| オプション操作 | Remove, Insert, Search, Sort の実装 | 可変（選択されたもののみ生成） |
| イテレーション | 要素走査の実装 | 可変（選択された場合のみ生成） |
| コントラクト適合 | 言語ネイティブのinterface/protocol実装 | 可変（言語×フィーチャーから導出） |

## 技術スタック

| カテゴリ | 技術 | 選定理由 |
|---------|------|---------|
| コード生成 | AI（Claude）によるテキスト生成 | DSLと構成の知識に基づき、言語を問わずコードを生成可能 |
| DSLバリデータ | Go | プロジェクトの主言語。YAML解析に適している |
| テスト | 各言語のプロパティベーステストFW | 生成先言語に応じて選択（Go: rapid、Python: hypothesis 等） |
