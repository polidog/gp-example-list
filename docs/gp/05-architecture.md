# アーキテクチャ設計: リストコンテナ

## アーキテクチャスタイル

### 選定したスタイル

**静的構成によるコード合成（Static Configuration with Code Composition）**

### 選定理由

本システムはジェネレーティブプログラミングに基づき、フィーチャー選択を**生成時（静的）に解決**してコードを生成する。ランタイムポリモーフィズムではなく、DSL仕様に基づいてAIが最適なコードを合成する方式を採用する。

- **可変性の解決タイミング**: すべての可変ポイントはコード生成時に確定する。生成後のコードには不要な間接層やランタイムオーバーヘッドが存在しない
- **多言語対応**: 生成対象が特定の言語に限定されないため、ランタイムパターン（Strategy, DI等）に依存せず、各言語のイディオムに沿ったコードを直接生成できる
- **学習目的への適合**: GPの本質である「仕様からの自動構成」を明確に体験できる

### 生成アーキテクチャの概要

```
┌─────────────────────────────────────────────┐
│                 DSL仕様書                      │
│  (フィーチャー選択 + パラメータ指定)            │
└─────────────┬───────────────────────────────┘
              │ 入力
              v
┌─────────────────────────────────────────────┐
│              ジェネレータ（AI）                 │
│  ┌─────────────────────────────────────┐    │
│  │         構成の知識                     │    │
│  │  (フィーチャー → コンポーネント対応)    │    │
│  └─────────────────────────────────────┘    │
│  ┌─────────────────────────────────────┐    │
│  │       実装コンポーネント群              │    │
│  │  (コードの構成要素・テンプレート)       │    │
│  └─────────────────────────────────────┘    │
└─────────────┬───────────────────────────────┘
              │ 出力
              v
┌─────────────────────────────────────────────┐
│            生成されたリストコンテナ              │
│  (指定言語の完全なソースコード)                  │
└─────────────────────────────────────────────┘
```

## 可変ポイントの実現方式

| 可変ポイント | 変化のタイプ | 実現方式 | 説明 |
|-------------|------------|---------|------|
| ElementType | パラメータ型 | 言語のジェネリクス機構 | 生成対象言語の型パラメータ機構（C++テンプレート、Javaジェネリクス、Go type parameter等）を使用。生成時に適切な構文を出力 |
| Ownership | 差し替え型 | コードフラグメント選択 | 3つの所有権バリアント（ExternalReference / OwnedReference / Copy）それぞれに対応するNode構造体とメモリ管理コードを持ち、仕様に応じて選択・合成 |
| Morphology | 差し替え型 | 型制約の切り替え | Monomorphic時は具象型制約、Polymorphic時はサブタイプ許容の型制約を生成。Ownership選択との組み合わせで適切なNode定義を決定 |
| LengthCounter | 差し替え型 | コードフラグメント合成 | 有効時: List構造体にカウンタフィールドを追加し、挿入・削除操作にカウンタ更新コードを織り込む。無効時: これらを省略 |
| CounterType | 差し替え型 | 型名の置換 | LengthCounter有効時、カウンタフィールドの型を short / int / long から選択して出力 |
| Tracing | 差し替え型 | コードフラグメント合成 | 有効時: 各操作の前後にトレースログ出力コードを織り込む。無効時: これらを省略 |

## 生成対象コードのモジュール構成

生成されるリストコンテナのコード構造:

```
生成コード
├── 型定義層
│   ├── Element型（ElementType + Morphology で決定）
│   ├── Node構造体（Ownership + Morphology で決定）
│   └── List構造体（共通 + LengthCounter/Tracing で拡張）
├── 操作層
│   ├── 基本操作（挿入・削除・走査・検索）[共通]
│   ├── メモリ管理（生成・破棄）[Ownership で決定]
│   ├── カウンタ更新 [LengthCounter 有効時のみ]
│   └── トレース出力 [Tracing 有効時のみ]
└── 公開API層
    └── リストインターフェース [共通]
```

### 各層の責務

| 層 | 責務 | 共通/可変 |
|---|------|----------|
| 型定義層 — Element型 | 要素の型表現を定義 | 可変（ElementType, Morphology） |
| 型定義層 — Node構造体 | リストの内部ノードを定義（データ保持方法と連結構造） | 可変（Ownership, Morphology） |
| 型定義層 — List構造体 | リスト本体の構造体を定義（head参照 + オプショナルフィールド） | 可変（LengthCounter, Tracing で拡張） |
| 操作層 — 基本操作 | 挿入・削除・走査・検索のアルゴリズム | 共通（ただし内部でOwnership/Morphologyに応じたコードが使われる） |
| 操作層 — メモリ管理 | ノード生成時・破棄時のメモリ確保/解放ロジック | 可変（Ownership） |
| 操作層 — カウンタ更新 | 挿入・削除時のカウンタインクリメント/デクリメント | 可変（LengthCounter有効時のみ存在） |
| 操作層 — トレース出力 | 操作前後のログ出力 | 可変（Tracing有効時のみ存在） |
| 公開API層 | 利用者に公開するメソッド/関数群 | 共通 |

## コード合成の戦略

### Ownership × Morphology の組み合わせ（Node構造体の決定）

Ownership と Morphology は Node 構造体に同時に影響するため、その組み合わせでNode定義が決まる:

| 組み合わせ | Node の data フィールド | 型制約 |
|-----------|----------------------|-------|
| Copy + Monomorphic | `T data`（値埋め込み） | T のみ |
| ExternalReference + Monomorphic | `T* data`（外部ポインタ） | T のみ |
| OwnedReference + Monomorphic | `T* data`（所有ポインタ） | T のみ |
| ExternalReference + Polymorphic | `T* data`（外部ポインタ） | T のサブタイプも可 |
| OwnedReference + Polymorphic | `T* data`（所有ポインタ） | T のサブタイプも可 |

### LengthCounter と Tracing の織り込み

LengthCounter と Tracing は基本操作に対する**横断的関心事（cross-cutting concern）** として、操作コードに織り込まれる:

```
insert(element):
  [Tracing有効時] trace("insert", element)
  node = create_node(element)     ← Ownership により実装が変わる
  link_node(node)                 ← 共通ロジック
  [LengthCounter有効時] length++
```

## 技術スタック

| カテゴリ | 技術 | 選定理由 |
|---------|------|---------|
| DSL形式 | YAML | 宣言的で人間が読みやすく、多くの言語でパーサーが利用可能 |
| ジェネレータ | AIプロンプト（Claude） | DSL仕様と構成の知識をプロンプトとして渡し、指定言語のコードを生成。多言語対応が自然に実現できる |
| 構成の知識 | Markdownドキュメント | AIが参照するルールとして、人間にも読める形式で記述。プロンプトの一部として活用 |
| 生成対象言語 | 任意（仕様で指定） | C++, Java, Go, Python, Rust 等。言語ごとのイディオムに沿ったコードを生成 |
